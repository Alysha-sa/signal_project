Explanation for each diagram:

1. Alert Generation System
Alert Generation System is the system that is responsible for generating warnings about the patient data and the identification of critical conditions that require immediate medical attention. The subsystem is based on the class AlertGenerator, which evaluates patient data using the class DataStorage. By separating the alert warning logic from data generation, the monitoring and analysis stay independent. This makes each component easier to manage and test by themselves.

The AlertGenerator uses DataStorage to gain access to patient history and real-time patient data. This design resembles a realistic hospital environment in which alerts are being activated by using stored patient data rather than using raw incoming signals. The evaluateData method uses the logic for determining whether the state of a patient meets the predefined alert criteria. If a criterion is met, an alert object is created and sent to the triggerAlert method that processes it.

The Alert class is structured in a way that it includes a patient identifier, and a description of the condition with a timestamp. This guarantees that every alert can be traced back, which means that this can be registered, sent, or processed by other components. Traceability is especially important in systems where safety is important like hospitals. Since in systems like this, accountability and frequent monitoring are required.

This diagram also includes a second AlertGenerator from the data generation package, that simulates alert events. This ensures a clear distinction between simulated alerts that are used for testing and real-time alerts that are triggered by a patient. This distinction prevents that testing logic from interfering with alert handling, which is an important boundary.


2. Data Storage System
The Data Storage System is responsible for safely storing, retrieving, and managing the patient data that is generated by the simulator. The main class is DataStorage, ehich is the bridge between patient identification and the list of patient data records. This structure makes it possible to efficiently retrieve all records assigned to a specific patient. The retentionPeriod attribute and deleteOldRecords method make sure that old data is automatically removed. This is important in a hospital environment where storage capacity is limited and the privacy regulations require that data is deleted after a certain period of time.

The PatientData class represents a record with a timestamp, that has a label and a value, like a heart rate reading at a specific time. By modelling each record like a separate object, the system supports a detailed retrieval and analysis of the patient history over time.

The DataRetriever class processes all demands that are being executed by medical staff or other subsystems. By separating the demanding logic from the storage logic, medical staff only communicates with the DataRetriever. This means that they do not have direct access to the underlying storage, which ensures the undelying storage cannot be modified, reducing the risk of unwanted changes in the data.

The AccessController class oversees the access by verifying if a specific role is authorized to get access to the data of a specific patient before the data is even retrieved. This is an important design decision when looking at healthcare, since sensitive patient data can only be accessed by authorized staff. In general, this subsystem prioritizes data integrity, privacy, and sustainability by a clear separation of responsibilities.


3. Patient Identification System
The Patient Identification System checks and makes sure that every incoming data point is correctly connected to the right patient in the hospital database. This is a requirement in the healthcare environment, since incorrect identification can lead to the wrong treatment desicions. This subsystem is based on three main classes: PatientIdentifier, IdentityManager, and HospitalPatient, with supporting MismatchLog for tracking errors.

The PatientIdentifier class is responsible for matching the incoming patient identifications from the simulator with the hospital records. The real search is assigned to IdentityManager, thus keeping a distinction between the matching logic and the data management. This distinction means that if, in the future, the matching algorithm changese, only the PatientIdentifier needs to be adapted, while the IdentityManager can stay untouched.

The IdentityManager acts as the main authority for patient identification within the system. It contains an overview of all registered patients and treats exceptional cases like mismatches, where an incoming ID does not correspond with any registered patient. Instead of silently ignoring similar errors, the system registers the roor via the MismatchLog class. This guarantees full traceability of identification erros, which is essential for debugging in safety-critical systems.

The HospitalPatient class saves and stores sensitive personal information, like names, date of birth, and medical history. These attributes are private, which means that these can only be accessed by controlled getter methods. This design reflects the importance of data privacy and prevents that unautorized parts of the system can access sensitive patient data directly.


4. Data Access Layer
The Data Access Layer is responsible for connecting external data sources with the rest of the system. The simulator data can stream using TCP, WebSocket, or file logs. So, this layer has to be flexible enough to process all three without having to change another subsystem. This can be achieved through the DataListener interface, which defines a common agreement that every input source must satisfy.

There are three concrete implementations provided: TCPDataListener processes incoming data from a TCP socket connection, WebSocketDataListener maintains real-time WebSocket streams, and FileDataListener reads data from log files. Every implementation manages its own connection internally, as a result of which the rest of the system stays completely clueless about the way data comes in. This follows a principle of which components at a higher lever are dependent on the DataListener instead of a specific input mechanism.

The DataParser class is responsible for regulating input strings to structured PatientData objects which the rest of the system can use. By keeping parsing logic in a seperate class, this design ensures that if data formatting changes, only the DataParser has to be updated. The method validateFormat adds an extra layer of sturdiness by rejecting incorrectly formatted input before it reaches the system.

The DataSourceAdapter acts as a bridge between incoming data and the storage layer. It receives parsed data and sends this to DataStorage, ensuring a smooth transfer between the access layer  and the storage system. Thanks to this design, when adding a new data source only a new DataListener needs to be added, without requiring changes somewhere else in the system.
