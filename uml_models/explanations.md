Explanation for each diagram:

1. Data Generation System

The Data Generation System is responsible for simulating cardiovascular health data for many patients. The PatientDataGenerator interface is the most essential part. This interface has a method called generate that takes a patientId and an outputStrategy. This helps keep things consistent and makes it easy to add types of health data generators without changing the old code.
Concrete implementations such as ECGDataGenerator, BloodPressureDataGenerator, BloodSaturationDataGenerator, and BloodLevelsDataGenerator each simulate a specific physiological signal. These classes maintain internal state (e.g., last recorded values or baseline measurements) to produce realistic, continuous variations over time rather than random, disconnected values. This reflects real-world patient monitoring where vital signs change gradually.
There are classes like ECGDataGenerator, BloodPressureDataGenerator and BloodSaturationDataGenerator that simulate physiological signals. These classes keep track of what happened like the last values or baseline measurements to make the data look real and change over time. This is like what happens in life when we monitor patients and their vital signs change slowly.
The HealthDataSimulator class orchestrates the entire subsystem. It manages patient IDs, schedules periodic data generation tasks using a ScheduledExecutorService, and ensures that each generator produces data at appropriate intervals. By separating scheduling logic from data generation logic, the system maintains clear responsibilities.
An important design aspect is the use of the OutputStrategy interface, which decouples data generation from data output. This allows the system to remain flexible and adaptable to different output mechanisms without modifying generator logic. Overall, this subsystem demonstrates modular design, separation of concerns, and scalability, which are essential for a real-time health monitoring environment.


2. Alert Generation System
The Alert Generation System is responsible for analyzing patient data and identifying critical conditions that require attention. This subsystem is revolves around the AlertGenerator class, which evaluates patient data taken from the DataStorage class. By separating alert logic from data generation, the system ensures that monitoring and analysis are and stay independent and maintainable.
The AlertGenerator depends on DataStorage to access the history or real-time records of a patient. This relationship reflects a realistic hospital setting where alerts are based on stored patient data rather than incoming signals. The evaluateData(Patient patient) method encapsulates the logic for determining whether a patient’s condition meets predefined alert criteria. If a condition is met, an Alert object is created and passed to the triggerAlert method.
The Alert class represents an alert that is structured and contains a patient identifier, condition description, and timestamp. This design guarantees that alerts are traceable and can be logged, transmitted, or acted upon by other components of the system.
Additionally, the system includes a second AlertGenerator in the data generation package, which simulates alert events. This shows a clear distinction between simulated alerts and real-time alerts. Overall, this subsystem emphasizes on the separation of concerns, clear responsibility boundaries, and the importance of traceability in safety-critical systems.


3. Data Input System
The Data Input System is responsible for overseeing how data enters the system and how it is stored for later use. This subsystem includes the DataReader interface and the DataStorage class. Together they provide a flexible and extensible way to handle incoming data.
The DataReader interface defines a method readData(DataStorage dataStorage) that allows different data sources to be combined into the system. For example, implementations could read data from files, TCP streams, or WebSocket connections. By using an interface, the system can easily support new input mechanisms without changing existing components, which is important for good software engineering processes.
The DataStorage class serves as a central system for patient data. It provides methods such as getRecords(start, end) to get stored data within a specific time range. This abstraction ensures that other parts of the system, such as the alert generation subsystem, do not need to know how the data is stored internally.
This separation between data input and data storage improves flexibility and maintainability. It also supports scalability, as different storage implementations could be introduced without affecting other subsystems. Overall, this subsystem plays a crucial role in allowing reliable data access and integration within the system.


4. Output System
The Output System is responsible for delivering generated patient data to various external destinations. This subsystem is built around the OutputStrategy interface, which defines a method output(patientId, timestamp, label, data). This design follows the Strategy pattern, allowing different output mechanisms to be used interchangeably without modifying the main system.
Several concrete implementations of this interface are provided, including ConsoleOutputStrategy, FileOutputStrategy, TcpOutputStrategy, and WebSocketOutputStrategy. Each implementation handles data output in a different way. For example, ConsoleOutputStrategy prints data to the console, while FileOutputStrategy writes data to files using a structured directory system. Network-based strategies such as TCP and WebSocket enable real-time data streaming to external systems or clients.
The use of an interface ensures that the HealthDataSimulator and data generators remain separated from the specifics of how data is transmitted or stored. This improves flexibility and allows the system to adapt to different environments or requirements, such as switching from local logging to real-time monitoring.
Additionally, this design supports scalability and extensibility. New output strategies can be added with minimal effort, without impacting existing functionality. Overall, the Output System demonstrates strong adherence to modular design principles and enables the system to work in diverse scenarios.
